App Kit Overview

App Kit Overview
The App Kit SDK lets you build apps for common payment and liquidity workflows
across blockchains. You can send, bridge, or swap tokens on supported blockchains in
just a few lines of code. It provides a type-safe interface that works with Viem, Ethers,
Solana Web3.js, and Circle Wallets. You can also extend the kit to support other wallet
providers and frameworks.

Key features
Simple setup: Perform App Kit actions in a few lines of code.
Application monetization: Collect a fee from end-users without writing new
code.
-​ Flexible configurations: Specify custom RPC endpoints and wallet clients.
-​ Broad compatibility: Works with Viem, Ethers, and Solana, integrating smoothly
with existing developer workflows.

App Kit Installation
To get started with App Kit, install the core package and an adapter that fits your
environment.
Note: App Kit requires a Kit Key obtained from the Circle Developer Console.

1.​ Install App Kit
Shell

# npm
npm install @circle-fin/stablecoin-kit
# yarn
yarn add @circle-fin/stablecoin-kit

2.​ Install your preferred adapter
Viem
Shell

# npm
npm install @circle-fin/adapter-viem-v2 viem
# yarn
yarn add @circle-fin/adapter-viem-v2 viem

Ethers
Shell

# npm

npm install @circle-fin/adapter-ethers-v6 ethers
# yarn
yarn add @circle-fin/adapter-ethers-v6 ethers

Solana
Shell

# npm
npm install @circle-fin/adapter-solana @solana/web3.js
# yarn
yarn add @circle-fin/adapter-solana @solana/web3.js

Circle Wallets
If you have a Circle Wallets account, you can use the Circle Wallets adapter to connect
to developer-controlled wallets and Circle Contracts.
Shell

# npm
npm install @circle-fin/adapter-circle-wallets
# npm yarn
yarn add @circle-fin/adapter-circle-wallets

Note: The Circle Wallets adapter requires an API Key and Entity Secret from the Circle
Developer Console. These credentials should never be exposed in the browser. For this
reason, the adapter is intended for server-side use only.

console.log("---------------Starting Sending---------------");
//Send to a different wallet address
} catch (err) {

}
};
void swapUSDTtoUSDC();

# Adapter Setups
App Kit supports these adapter types:
* **Chain client adapters** abstract away the complexity of different blockchain
architectures. Each adapter manages blockchain-specific fund orchestration
operations.
* The **Circle Wallet adapter** is available if you already use Circle's
[developer-controlled wallets](/wallets/dev-controlled). It uses your Circle
Wallets account to let you transfer between
[supported blockchains](/app-kit/references/supported-blockchains).
## Chain client adapter setups
App Kit works with these client adapter libraries:
* [`viem`](https://viem.sh/) v2 for EVM-compatible blockchains
* [`ethers`](https://ethers.org/) v6 for EVM-compatible blockchains
* [`solana`](https://solana.com/) for the Solana blockchain
You only need to set up with one EVM adapter, either Viem or Ethers. Add the
Solana adapter to your setup only if you plan to transfer to or from Solana.
The following table shows ways to set up chain client adapters and when to use
each setup.
| Setup method
| Description
| Use case
|
| --------------------------------- |
---------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| [Standard](#standard-setup)
| Standard setup that creates an adapter from a
private wallet key. Uses default built-in public RPC endpoints and factory functions. |
Testing environments that need a quick start without custom RPC configuration. |

| [Custom RPC](#custom-rpc)
| Private key setup that lets you configure an RPC
endpoint.
| Production deployments that
require reliable RPC providers.
|
| [Browser wallet](#browser-wallet) | Uses an injected wallet provider, such as
MetaMask or Phantom, to create an adapter in the browser.
|
Browser-based apps that rely on user wallet providers.
|
### Standard setup
This setup is the fastest way to start. Create one adapter from your wallet
private key to transfer tokens between EVM-compatible blockchains. Add the
Solana adapter to transfer tokens to or from Solana.
This setup uses public RPC endpoints and factory functions. For production, you
should configure a [custom RPC](#custom-rpc). Public connections have rate
limits and might be slow.
Select the tab to see code examples for your adapter choice.

### Custom RPC
You can replace the public RPC from the [standard setup](#standard-setup) with
your own connection. For production, you should use a paid service like
[Alchemy](https://www.alchemy.com/) or [QuickNode](https://www.quicknode.com/).
These services are more reliable than free connections.
The following code shows how to set up your own RPC. Pick your adapter to see
the code.
<Tabs>
<Tab title="Viem">
To use your own connection, replace the default one and add your custom RPC
endpoints. Map each chain to its RPC endpoint so one adapter can work across
multiple chains. This example uses Alchemy connections:
```typescript TypeScript theme={null}
import { createViemAdapterFromPrivateKey } from "@circle-fin/adapter-viem-v2";
import { EthereumSepolia, ArcTestnet } from "@circle-fin/bridge-kit/chains";

import { createPublicClient, http } from "viem";
// Map RPC endpoints by chain name
const RPC_BY_CHAIN_NAME: Record<string, string> = {
[EthereumSepolia.name]:
`https://eth-sepolia.g.alchemy.com/v2/${process.env.ALCHEMY_KEY}`,
[ArcTestnet.name]:
`https://arc-testnet.g.alchemy.com/v2/${process.env.ALCHEMY_KEY}`,
};
// Create an adapter
const adapter = createViemAdapterFromPrivateKey({
privateKey: process.env.PRIVATE_KEY as string,
// Replace the default connection
getPublicClient: ({ chain }) => {
const rpcUrl = RPC_BY_CHAIN_NAME[chain.name];
if (!rpcUrl) {
throw new Error(`No RPC configured for chain: ${chain.name}`);
}
return createPublicClient({
chain,
transport: http(rpcUrl, {
retryCount: 3,
timeout: 10000,
}),
});
},
});
```
</Tab>
<Tab title="Ethers">
To use your own connection, replace the default one and add your custom RPC
endpoints. Map each chain to its RPC endpoint so one adapter can work across
multiple chains. This example uses Alchemy connections:
```typescript TypeScript theme={null}
import { createEthersAdapterFromPrivateKey } from "@circle-fin/adapter-ethers-v6";
import { EthereumSepolia, ArcTestnet } from "@circle-fin/bridge-kit/chains";
import { JsonRpcProvider } from "ethers";

// Map RPC endpoints by chain name
const RPC_BY_CHAIN_NAME: Record<string, string> = {
[EthereumSepolia.name]:
`https://eth-sepolia.g.alchemy.com/v2/${process.env.ALCHEMY_KEY}`,
[ArcTestnet.name]:
`https://arc-testnet.g.alchemy.com/v2/${process.env.ALCHEMY_KEY}`,
};
// Create an adapter
const adapter = createEthersAdapterFromPrivateKey({
privateKey: process.env.PRIVATE_KEY as string,
// Replace the default connection
getProvider: ({ chain }) => {
const rpcUrl = RPC_BY_CHAIN_NAME[chain.name];
if (!rpcUrl) {
throw new Error(`No RPC configured for chain: ${chain.name}`);
}
return new JsonRpcProvider(rpcUrl);
},
});
```
</Tab>
<Tab title="Solana">
To use your own connection, create a custom connection to your RPC endpoint.
Point to that connection when making a Solana adapter. This example creates a
custom connection using Alchemy:
```typescript TypeScript theme={null}
import { createSolanaKitAdapterFromPrivateKey } from
"@circle-fin/adapter-solana-kit";
```

--## Event Types
Bridge events are prefixed with `bridge.` to namespace them within StablecoinKit:
| Event | Description |
|-------|-------------|
| `bridge.approve` | Token approval transaction completed |
| `bridge.burn` | Source chain burn transaction completed |
| `bridge.attestation` | CCTP attestation received |
| `bridge.mint` | Destination chain mint transaction completed |
| `*` | Wildcard listener for all events |
**Event Payload**
Each event handler receives a payload with:
- `method` - The action name (e.g., `'approve'`, `'burn'`)
- `values` - Action-specific data including `txHash`, `explorerUrl`, etc.
**Usage Example**
```typescript
const kit = new StablecoinKit()

// Track progress through bridge steps
kit.on('bridge.approve', (payload) => {
console.log('Step 1/4: Approval complete')
console.log('Tx:', payload.values.txHash)
})
kit.on('bridge.burn', (payload) => {
console.log('Step 2/4: Burn complete')
console.log('Tx:', payload.values.txHash)
})
kit.on('bridge.attestation', (payload) => {
console.log('Step 3/4: Attestation received')
})
kit.on('bridge.mint', (payload) => {
console.log('Step 4/4: Mint complete')
console.log('Tx:', payload.values.txHash)
})
// Or use wildcard to log all events
kit.on('*', (event) => {
console.log(`[${event.method}]`, event.values)
})
// Execute bridge
await kit.bridge({
from: { adapter, chain: 'Ethereum' },
to: { adapter, chain: 'Base' },
amount: '100.00'
})
```

Set Slippage Tolerance or Stop Limit

Set Slippage Tolerance or Stop Limit
Set a slippage tolerance or stop limit to avoid receiving less than expected due to
market fluctuations:
●​ A slippage tolerance lets you set the maximum percentage difference you’re
willing to accept between the estimated and actual swap amount, expressed in
bps (for example, 100 bps = 1%).
●​ A stop limit lets you set the exact minimum amount you want to receive.
If both are configured, the stop limit takes precedence.

Set slippage tolerance
This example sets a slippage tolerance of 100 bps (1%):
JavaScript

const output = await kit.swap({
from: { adapter, chain: "Ethereum" },
tokenIn: 'USDT',
amount: '1.00',
tokenOut: 'USDC',
config: {
// Set the splippage tolerance to 100 bps
slippageBps: 100
}
})

With `slippageBps` as `100`, you receive at least 99% of the estimated amount. For
example, if the swap rate is 1 USDT to 1 USDC, you receive at least 0.99 USDC.
Note: The `slippageBps` default is 300 bps. `0` represents no tolerance which increase
transaction failure rates.

Set stop limit
`stopLimit` defines the minimum amount of the token type specified in `tokenOut` you
receive on a swap.

This example sets a stop limit of 0.95 USDC, meaning you won’t receive any less than
that on a swap:
JavaScript

const output = await kit.swap({
from: { adapter, chain: "Ethereum" },
tokenIn: 'USDT',
amount: '1.00',
tokenOut: 'USDC',
config: {
// Set stop limit to 0.95 USDC
stopLimit: '0.95',
}
})

Limiting slippage
Slippage is the difference between the expected price of a trade and the price at which
the trade is actually executed. You can optionally limit the slippage in bps via the
following:
JavaScript
// Default slippage 300 bps
slippageBps?: 10

Default slippage is 300 bps. Setting slippageBps to 0 means no tolerance, which can
lead to higher failure.
Alternatively you can specify stopLimit as the min amount of tokenOut you would like to
receive. If both are provided, the function will respect stopLimit and ignore slippageBps.
JavaScript
stopLimit?: '99.5',

Allowance strategy
Allowance strategy applies to ERC20 tokens. It configures the amount of tokenIn
allowed to be used in a contract call, in this case swap. When not provided, we will
default to a suitable strategy to request permit if it’s not already granted. You can
optionally provide it for the following setups:
●​ null: Do not do any allowance actions
●​ approve-exact: Approves the exact amount needed for the action
●​ approve-max: Approves the max amount (max uint256)
●​ permit-exact: Sign a permit for the exact amount needed for the action
●​ permit-max: Sign a permit for the max amount (max unit256)

JavaScript
allowanceStrategy?: undefined | null | 'approve-exact' | 'approve-max' |
'permit-exact' | 'permit-max'

Swapping with native token or on Solana does not require allowance strategy
configuration.

Token aliases supported
We support the following token aliases, for all other tokens you can always specify the
contract address:
●​ USDC
●​ EURC
●​ USDT
●​ ETH

Provider decision
This doc captures the detailed decisions we have to make:
the following:

SWAP option , including

1.​ In our vision, is it better to integrate with DEX aggregators, or DEXs directly?
a.​ Need to review with security on risks of integrating with either, SOC2
compliance implications taking dependencies on 3rd party permissioned
services.
b.​ Does DEX aggregator indeed provide better liquidity and prices as they
claim to? And at what volume, or trading pairs like USDT to USDC?
2.​ If we choose to go with DEX aggregator, we will need a micro-service that
proxies the underlying DEX aggregator(s) with our fee configuration, and open a
permissioned API endpoint for StablecoinKit. Permissionless access opens up
too many risks.
3.​ We will need to include fee taker capability. Depends on whether we go with
DEXs or DEX aggregators, the integration would be different:
a.​ To integrate with DEXs, we need to integrate fee taker with Swap so it’s an
atomic action (caveats of separate calls captured in the doc above)
b.​ To integrate with DEX aggregators, fee can be specified as part of the
DEX aggregator API calls.

* Why use account.swap() (all-in-one pattern)?
* - Simplest developer experience - one function call
* - Automatically handles quote creation and execution
* - Manages Permit2 signatures transparently

* - Built-in error handling for common issues
* - Best for 90% of swap use cases
*
* This example shows two approaches:
*
* Approach 1: All-in-one pattern (RECOMMENDED)
* - Uses account.swap(...) with inline options
* - Creates and executes swaps in a single call
* - Automatically validates liquidity and throws clear errors
* - Minimal code, maximum convenience
*
* Approach 2: Create-then-execute pattern (advanced)
* - First creates a swap quote using account.quoteSwap()
* - Allows inspection of swap details before execution
* - Provides more control for complex scenarios
* - Use when you need conditional logic based on swap details

provided by Circle. Following the steps below to create a Kit Key:

1.​ Create a Circle Developer Console account
Enter the Circle Developer Console sign-up page to create an account. Skip to the
sign-in step if you have an existing account.

2.​ Sign in to the Circle Developer Console

3.​ Go to Keys section

4.​ Click on Create a Key

5.​ Select Kit Key and provide Key Name to create a key

6.​ Copy and save the Kit Key
Copy and save the Kit Key securely; you will NOT be able to view the key content
again.

You will be able to manage the created Kit Key in the Keys section once creation is
done.

7.​ Configure the Kit Key in your application
